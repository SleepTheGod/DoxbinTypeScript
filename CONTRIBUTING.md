# Contributing to Doxbin

Thank you for your interest in contributing to Doxbin! This document provides guidelines for contributing to the project.

## Code of Conduct

- Be respectful and professional
- Follow the existing code style
- Write clear commit messages
- Test your changes before submitting

## Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/YOUR_USERNAME/DoxbinTypeScript.git`
3. Create a branch: `git checkout -b feature/your-feature-name`
4. Make your changes
5. Test thoroughly
6. Commit: `git commit -m "Add: your feature description"`
7. Push: `git push origin feature/your-feature-name`
8. Create a Pull Request

## Development Guidelines

### Code Style

- Use TypeScript for all new code
- Follow the existing code structure
- Use meaningful variable and function names
- Add comments for complex logic
- Format code with Prettier before committing

### Commit Messages

Use conventional commit format:
- `feat:` New feature
- `fix:` Bug fix
- `docs:` Documentation changes
- `style:` Code style changes (formatting, etc.)
- `refactor:` Code refactoring
- `test:` Adding tests
- `chore:` Maintenance tasks

Example: `feat: add search functionality to paste list`

### Testing

- Test all new features thoroughly
- Test on both desktop and mobile
- Verify database operations work correctly
- Check for XSS and SQL injection vulnerabilities

### Pull Request Process

1. Update README.md if needed
2. Ensure all tests pass
3. Update documentation
4. Request review from maintainers
5. Address review feedback
6. Wait for approval and merge

## Feature Requests

Open an issue with:
- Clear description of the feature
- Use case and benefits
- Implementation suggestions (optional)

## Bug Reports

Open an issue with:
- Clear description of the bug
- Steps to reproduce
- Expected vs actual behavior
- Screenshots if applicable
- Environment details (OS, browser, etc.)

## Questions

For questions, open a discussion or reach out on Telegram.

Thank you for contributing!
